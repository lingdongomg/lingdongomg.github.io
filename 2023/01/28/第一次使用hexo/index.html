第一次使用hexo

这段时间在干嘛

在去年的今天，我利用学生账号购买了一年的腾讯云轻量级服务器，当时只花了五十块钱左右，再花了十块钱买了hukaichao.cn的域名。

当时在完成了服务器、域名的备份之后，在22年2月份利用halo框架创建了属于自己的博客，算是自己的第一个博客。

时间过的好快，眨眼一年过去了，之前买的服务器也到期了，如果继续续费一年，那要两三百块，太不值得了。我一年来除了用那个服务器当个人博客，也就挂一下打卡签到的脚本，现在疫情解封，不再需要打卡了，花大价钱买服务器感觉有点亏。

在几大云服务器厂商转了一圈，腾讯云我已经不是新用户，买服务器都要近两百块，阿里云还是新用户，利用学生身份可以白嫖七个月服务器，可以凑合用。另外看了国外的，之前利用GitHub学生认证有一个开发者工具箱，里面赠送了很多东西，其中就有微软的Azure提供了100美元额度的使用权，可是我不知道怎么触发风控怎么也用不了，可能是我注册邮箱不是学校的edu邮箱吧（小气学校不给edu邮箱，当时GitHub学生认证都花了好长时间）。

同时也有DigitalOcean提供一百美元使用权，不过得要信用卡注册账号，我没有（学生好难申请信用卡，招商银行以前有专门给学生的信用卡申请，现在都停了，必须在职人员才受理哎）。处处受挫，不过好在Namecheap可以用，在他这白嫖了一年的.me域名，就是hukaichao.me。感觉还不错，花了一天时间弄SSL证书（这个也是GitHub工具包赠送的），把它安装好，然后研究了一下博客系统，其实可以用阿里云服务器搭配这个域名继续用halo的博客，不过我感觉这个总归用不长久，服务器过期了又舍不得续费，最后决定用GitHub+hexo的静态化博客页面，用着其实挺不错，主要是能永久用，不担心过期。

hexo 和halo

最开始选用halo是因为买了服务器之后才开始研究博客系统，看上了halo的后台管理和主题界面，说实话用起来确实很方便，写博客直接在后台写，插入图片，添加标签什么的都很友好，并且开发社区很活跃，最开始采用的宝塔面板傻瓜式点击部署的，后面服务器被人入侵过一次（入侵原因是我照着文档部署的时候新建用户采用了halo文档里的默认密码…），在我服务器中留下了什么文件，我发现后将服务器重置了，第二次部署的时候发现采用docker更方便，即使一点代码都不懂也能按照文档顺利使用。

这次选择用hexo是发现很多大佬也采用的hexo，并且hexo可以很好和GitHub的pages配合，不需要服务器也能将静态页面部署到网页上，确实很方便，不过功能上感觉不如halo方便。比如我现在写文档就是在本地编辑md文件，采用的软件是Typora，这个软件其实也挺好的，但是插入图片很难受。如果我想要插入一张图片，我复制到Typora中后还需要将图片另存到hexo的resource文件夹下才能在web端正常访问，不过Typora倒是可以设置自动复制，然后我再改一下文档的图片根路径，其实用着也还算流畅。

今天选主题改配置文件什么的也花了不少时间，在写完文章用hexo d命令上传的时候会报SSL连接错误，查了一下是本地git没有和GitHub建立远程连接导致，以及之前的访问方式是https，现在改成了git@github.com:lingdongomg/，这种好像是SSH远程连接。这两点修改之后偶尔还出现SSL连接错误，后面发现再上传前输入hexo clean清理一下缓存好像就好很多。

DNS

在博客弄好之后发现了个问题，那就是每次本地push内容到GitHub后，我博客的域名都会重定向到lingdongomg.github.io，很伤脑筋。百度后发现是CNAME文件没有，后面添加了CNAME后确实不会重定向了，但之前能顺利访问的hukaichao.me在edge浏览器访问不了了，域名没有SSL安全认证，之前设置的又只能通过https访问（GitHub设置的，即使访问http也会重定向到https）。不过这个时候手机浏览器可以正常访问。

睡了一晚后这个问题依然存在，就开始找途径解决。我在namecheap上的这个域名是有SSL安全证书的，证书还发我邮箱保存在我本地了，但是我怎么用呢？去年的那个hukaichao.cn的SSL证书是腾讯云免费给的，当时输入了个人信息就申请到了，同样将证书下载到本地之后通过宝塔的面板将密钥和证书内容复制上去保存后点击部署就可以了，毫无疑问证书要部署到服务器端。

别人访问我的博客的大概流程应该是：在地址栏输入我的域名->在域名的DNS映射表（在域名的服务商那里设置）中将域名转化为公网ip地址->通过服务器的公网ip地址搭配默认端口号访问到我的服务器对外开放的web界面。那现在我要将静态博客部署到GitHub pages上，怎么处理呢？

首先在域名的DNS映射表中添加两条A Record，像下面这样：

Host同时设置@和www两种，应该是两种访问方式，具体有什么用我搜不到。我之前只用了其中一种好像不行。

Value对应的ip地址是GitHub的公网IP，有四个，这里只用选其中一个就行。

A Record的功能就是将hukaichao.me的域名解析为GitHub的ip地址

在配置完这两者之后再加一条CNAME Record

CNAME Record的作用是将value中的lingdongomg.github.io这个域名重定向到hukaichao.me域名上，这样访问lingdongomg.github.io这个网址也会跳到我的博客上。

到这里完成了DNS的配置，在GitHub中能顺利通过DNS的检验了。然后之前出现的问题就消失了，我本地的SSL证书并没有起到任何作用，但我的博客就是能通过https访问了。这里可能是GitHub Pages自动干的，其中的原因得从异常信息来分析。

现在想起来的异常信息有：Both hukaichao.me and its alternate name are improperly configured Domain does not resolve to the GitHub Pages

这个的原因就是上面的DNS表的A Record没设置好，在设置好后就没有这个异常了。

在这个表设置好后在Github Pages设置页的自动颁发TLS证书时出现过这个异常信息：Certificate Request Error: Certificate provisioning will retry automatically in a short period, please be patient.

这个问题查阅之后得到的解决方案是就是在上面DNS映射中增加lingdongomg.github.io的CNAME Record

想起来我之前确实没加，总之所有问题的确都归结于DNS，在配置好后万事大吉

之前博客内容转移

之前一年博客多多少少页有六十来篇文章，我得将其从halo转移到hexo，好在halo备份很方便，还能直接下载md文件，在下载完成之后将md文件和图片放在指定目录就完成转移了。

不过遇到了点些警告信息：warning: in the working copy of ‘tags/面经-笔经/index.html’, LF will be replaced by CRLF the next time Git touches it

这个警告信息的原因可能是之前我的halo博客是部署到Linux服务器上，其中的换行的格式是LF，现在我用typora写博客是在windows上，换行格式是CRLF，所以会有这个警告，虽然看着难受不过没什么影响，也可以修改设置，这个能查到很多信息，在此就简单记录一下。

结语

之前因为服务器到期，很长一段时间没更新博客，其实也有在学东西，但是没有记录，后面回想起来的时候总感觉自己没学什么内容，学了的一些东西也感觉忘的差不多了，还是得写博客，当作日志和学习记录，回顾和查资料都会方便很多，现在要开始勤更新博客了。遇到和解决问题都会写在博客里面。

第一篇博客暂且唠这些吧，后面开始更新前段时间学习的东西和现在正在学习的东西。
