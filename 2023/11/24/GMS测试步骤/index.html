GMS测试步骤

GMS测试项目

1、测试环境

1.1 Ubuntu端配置（示例）

  • 系统版本：Ubuntu 18.04.6 LTS
  • JDK版本： 11.0.20
  • SDK中build-tools版本：33.0.2
  • ADB版本：1.0.41
  • cpu：i5-12400
  • 内存：16G（这个很重要，在8G的情况下极其容易中断测试）
  • 硬盘：512G固态

​

​ 除了Ubuntu之外还需要一台 windows 系统的PC，用于烧录固件、SN码还需要装 platform-tools （需要用fastboot：34.0.4）

​ 此外需要一个能连接外网的路由，Ubuntu和待测设备均需连接外网。

1.2 Android端配置

  • 将设备恢复出厂设置：设置 > 备份和重置 > 恢复出厂数据。

  • 将设备的语言设置为英语（美国）：设置 > 语言和输入法 > 语言。

  • 如果设备上有 GPS 或 Wi-Fi/蜂窝网络功能，请打开位置设置：设置 > 位置 > 开。

  • 将屏幕亮度设置为最低，为了省电

  • 打开蓝牙

  • 连接到支持 IPv6 的 Wi-Fi 网络，可以将 DUT 视为独立客户端（请参阅上面的物理环境），并且具有 Internet 连接：设置 > Wi-Fi 。

  • 确保设备上未设置锁定图案或密码：设置 > 安全 > 屏幕锁定 > 无。

  • 将时间设置为 12 小时格式：设置 > 日期和时间 > 使用 24 小时格式 > 关闭。

  • 在设备上启用USB 调试：设置 > 开发者选项 > USB 调试。

  • 将设备设置为保持唤醒：设置 > 开发者选项 > 保持唤醒 > 开启。

  • 在Android 4.2 或更高版本中，关闭 USB 应用程序验证：设置 > 开发者选项 > 通过 USB 验证应用程序 > 关闭。

  • 在Android 13 或更高版本中，将设备设置为允许模拟调制解调器：设置 > 开发者选项 > 允许模拟调制解调器 > 开。

  • 仅在 Android 5.x 和 4.4.x中，将设备设置为允许模拟位置：设置 > 开发人员选项 > 允许模拟位置 > 开。

  • 设置手机连接外网（单测某些模块可不设置）

  • 插入 SIM 卡，打开移动数据 （单测某些模块可不设置）

  • 启动浏览器并关闭任何启动/设置屏幕。

  • 如果设备支持自定义默认字体，请将默认sans-serif字体系列设置为Roboto （AOSP 版本中使用的默认sans-serif字体系列）。

  • 连接待测设备和 PC，在 PC 端执行 adb devices，确保连接成功

2、测试指令

CTS、GTS、VTS、GSI、STS测试所执行的指令基本一致，因此放在一起，便于查阅

启动测试

  • 手机连接到电脑
  • 打开 Ubuntu，切换到 需要测试项目对应 路径
  • 进入 测试项目 控制台： ./tools/cts-tradefed
  • 运行 对应项目 的测试命令 ： run cts、 run gts、 run vts 、 run cts-on-gsi、 run sts-dynamic-incremental
    • 如果是多台设备，需要携带参数--shard-count number_of_shards （Android9及以上）
    • 如果是指定设备，需要采用-s deviceID 指定，多个设备需要多次 -s
    • 重新执行之前失败和未执行的测试指令：run retry -r sessionID，其中的sessionID可以通过指令l r拿到。测试基本都是多轮，一般需要执行到连续3次结果一致，此时再查看结果。在进行的差不多的时候可以将所有设备恢复出厂设置，再继续测。
    • 在拿到结果后，分析其中产生的原因，再进行单独测试，此时可以使用 --exclude-filter test_module_name 来进行过滤一些费时的测试，将其从运行中排除，例如CTS中的location和network方面测试。具体指令示例:run retry -r 66 --shard-count 6 -s TVE8313M0001 -s TVE8313M0002 -s TVE8313M0003 -s TVE8313M0004 -s TVE8313M0005 -s TVE8313M0006 --exclude-filter CtsHostsideNetworkTests --exclude-filter CtsLibcoreTestCases --exclude-filter CtsMediaCodecTestCases --exclude-filter CtsMediaPlayerTestCases

常用命令

  • 列出所有测试结果： l r
  • 流出所有检测到或已知的设备： l d
  • 查看正在运行的设备状态： l i
  • 查看所运行的指令：l c
  • 单跑某一个模块：run cts -m <模块名>
  • 单跑某个测试项：run cts -m <模块名> -t <测试项名>
  • 单跑某个测试项里具体的一项：run cts -m <模块名> -t <测试项名>#<测试项中的某功能测试项>
  • 复测指令：run retry -r <序列号session>

测试结果

测试的结果文件在 results 目录下，找到对应日期时间的文件夹，该文件夹就是所需要的结果文件，要交付时将这个文件夹打包成压缩包交付。这里面有test_result_failures_suite.html和test_result_html，主要看的是前者，里面全是失败项的详情。

如果要详细知道某一项为什么不通过，可以进行单测这一项然后查看日志，日志在logs文件夹中。

3、CTS

3.1 CTS介绍

Compatibility Test Suite 意为兼容性测试套件，是 Google 推出的 Android 平台兼容性测试机制。CTS 测试主要是测试 OEM 厂商设计的 Android 平台是不是符合 Android 的 API 接口定义。通过 CTS 测试不仅可以保证 Android 设备上正常运行，同时，也能够让所有的开发者放心地制作高质量的应用程序，因此只有通过 CTS 认证的设备才能合法的安装使用 Google market 等 Google 应用。

4、CTS Verifier

4.1 CTS Verifier介绍

CTS Verifier 算是CTS的一部分，需要手动进行，主要用于测试那些自动测试系统无法测试的功能，比如相机、传感器等。由于硬件配置或其他原因，不同手机上部分测试项目被隐藏，也就是说CTS Verifier中case的总数，取决于测试机支持哪些功能。

测试

  • PC 端执行如下指令：adb shell settings put global hidden_api_policy 1

  • 安装 CtsVerifier.apk : adb install -r -g CtsVerifier.apk

  • CtsVerifier的权限全部授予，语言也要切成英文

  • 看CtsVerifier测试文档，进行具体操作

  • 最后要输入日志需要如下指令

    • adb shell appops set com.android.cts.verifier android:read_device_identifiers allow
    • adb shell appops set com.android.cts.verifier MANAGE_EXTERNAL_STORAGE 0

  • 查看cpu架构指令：adb shell getprop ro.product.cpu.abi

tip：如果电脑连接了多台设备，在执行adb时可以在adb后通过 -s SerialNumber 进行指定设备执行，例如adb -s TVE1070M001 -s TVE1070M002 install -r -g CtsVerifier.apk

5、VTS

5.1 VTS介绍

Vendor Test Suite 意为供应商测试套件。以前Android的系统升级是很麻烦的，为了能更快的将设备升级到新的Android版本，Android O 开始新引入了 Project Treble，Project Treble 适用于搭载 Android O 及后续版本的所有新设备。Android 7.x 及更早版本中没有正式的Vendor层接口，因此每次更新系统都相对耗时和困难。Android O 之后，Treble 提供了稳定的Vendor层接口，供设备制造商访问 Android 代码中特定于硬件的部分，这样就可以只更新框架层，减少升级系统带来的成本和困难。为了确保Vendor层实现的前向兼容性，新的Vendor层接口会由供应商测试套件 (VTS) 进行验证，该套件类似于兼容性测试套件 (CTS)。

5.2 VTS测试

替换谷歌system.img

VTS 测试要求刷入谷歌提供 AOSP 的 system.img (GSI) 。在 user 版本中，如果直接使用 flash tool 单独烧录GSI时，会导致无法开机。之所以会出现这种问题，是由于在 user/userdebug 版本中， dm-verity 是使能的，替换 GSI 后导致 dm-verity 不能通过。 如果要解决这个问题，就需要进行 unlock 操作，并且要用 fastboot 来刷入谷歌提供的 system.img 。

烧录具体操作如下：

烧录时推荐采用windows，如果烧录在fastboot reboot fastboot这一步卡住，windows的问题是驱动未更新，需要手动到设置-更新和安全-查看可选更新，更新对应平台和安卓的驱动。如果是Ubuntu卡住，则可能是 fastboot 版本问题。

  • 在设置中打开System -> Developer options -> OEM unlocking 选项

  • 在设置中打开 USB debugging 选项

  • 手机连接到电脑上，输入 adb devices 命令确认 adb 可以正常使用

  • 进入 fastboot 模式。方法一： adb reboot bootloader；方法二：长按音量 + 和电源键

  • 执行命令 fastboot flashing unlock ，执行命令后需要选择音量 + ，确认设备已经 unlock

  • 执行命令fastboot getvar unlocked 确定设备 unlock 成功

  • 擦除数据分区： fastboot -w

  • 擦除system.img 分区： fastboot erase system

  • 刷入user固件中的 vendor_boot 和 preloader_a （如果是刷 GSI 则不需要这步）

  • 将 Android 设备重新启动到 Fastboot 模式fastboot reboot fastboot

  • 执行 fastboot 命令刷入对应的 **system.img (在GSI文件夹下)**：

  • 重启： fastboot reboot

7、STS

7.1 STS介绍

Android Security Test Suite (STS). 是谷歌关于android安全补丁安装情况的一个测试套件STS和security patch相关的，是CTS测试新增加一项安全测试套件。STS是201808才开始测试的。Security patch日期在3个月内是GTS的一个case，如果不通过无法获得google认证。在18年5月之前谷歌对于security patch这个属性都是在build库下面跟着aosp更新的。现在需要通过STS之后由vendor来更新。STS需要通过userdebug版本的targetfile来生成一个user版本的结果。

7.2 STS测试流程

STS测试需要将待测设备烧录debug固件，debug固件一般存放在user固件旁边的文件夹中

9、机器烧录

系统：windows

在初次使用时需要安装驱动，文件为安装文件/烧录工具/DriverInstall.exe

9.1 固件烧录

工厂直接发过来的机器固件和SN码都不是最新的，一般拿到手时需要自己刷上最新的固件和SN码

拿到固件后，通过固件烧录工具进行烧录，这里固件烧录工具是Smart Phone Flash Tool，kernel版本为4.19及以下工具为V5，4.19以上工具为V6。查看kernel的方式：adb shell cat /proc/version

该工具如上图，左上角有一个Auto Reboot 可以选上，用途是烧录完成自动重启设备。

右上角的Download-XML 需要选中固件文件夹中的 images/download_agent/flash.xml ，选中后下方的Downlad会出现许多项目，将升级方式选为 Firmware Upgrade，该升级方式是升级，效果为保留SN码，将其他内容格式化，然后进行升级。其他还有两个选项为全部格式化，和仅下载文件。

在选完上述内容后点击Download，再将关机状态下的设备用数据线连接上电脑，即可自动开始烧录固件。

版本V5稍微有些不同，选取文件时不是选取的 Download-xml，而是Scatter-loading File，这里是选取一个txt文件，在固件的image目录下会有一个XXXXXX_Android_scatter.txt文件。

9.2 SN码烧录

SN码烧录工具为SN Writer V1.2152.0.0，界面如下，这个工具每一个项目对应的版本都不一样。在打开时可能会遇到下面问题：

这个问题是电脑未安装32位的vc++程序包导致的，安装一下就好了，文件为vcredist_x86.exe

烧录时先点击上面的Key Files，选中对应版本的MTK HDCP/DRM KEY，这个是要选择文件夹，一般是drmkey文件夹，这里的文件夹是每个项目对应的，不在固件中，我使用时是别人直接发我的。

然后选择attestation Key，这个是选择一个googlekey的bin文件，一般是在drmkey文件夹外面的bin文件，而不是drmkey里面的。

设置完这两个后再点击上面图片中的System Config，选中第一个 Barcode，就是烧录sn码了，其他的选项大部分是通讯设备用的，WiFi和蓝牙一般也不需要。System Config 界面的最下方需要选一个AP_DB文件，该文件是取自固件中的 images文件夹下的 APDB_CPU型号_固件版本号 文件，有多个话选择版本号数字最大的那个。设置完成点击SAVE，就可以点击START，此时要求输入SN码，SN只能字母(无论大小写)+数字；比如1070的项目，可以写：TVE1070M00001；sn的长度最好控制在8-15个字符。

SN码输入完成后将关机状态的设备连接数据线插上电脑，会自动开始烧录，烧录完不会自动开机，需要手动开机，然后连接adb验证是否烧录成功。
