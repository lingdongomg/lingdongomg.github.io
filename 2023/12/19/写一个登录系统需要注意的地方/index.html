写一个登录系统需要注意的地方

登陆系统是大多数系统都有的，其实现上不难，完成注册和登录即可，但要想写一个功能齐全能在生产环境使用的登录系统，要考虑的内容还是有很多的。本文基于RuoYi-Plus框架记录一下它的登录系统的实现。

注册登录

加密

登录系统最重要的莫过于对密码进行加密，密码是严禁采用明文的形式存储到数据库中的，加密算法一般为BCrypt，加密部分可以见文章 “密码学学习”

保存登录状态

一个系统不可能每次使用都要重新登录，不然会给用户带来极差的体验，所以一般都会保存用户的登录状态，让用户第二次打开的时候可以免去登录直接使用，根据系统的安全性来设计登录状态的保存时间，保存的具体方式一般为两种：基于token的验证和基于session的验证。

token

在许多场景中，术语 “token” 可以指代不同类型的令牌，包括但不限于：

  1. JWT（JSON Web Token）： 是一种使用 JSON 格式编码的 token，通常包含有关用户身份和其他声明的信息。JWT 具有自包含性，可以在多个服务之间传递，用于实现身份验证和授权。
  2. Access Token（访问令牌）： 通常指的是 OAuth 2.0 中的访问令牌，用于授权客户端访问受保护资源。Access Token 可以是 JWT 格式的，也可以是其他格式。
  3. Refresh Token（刷新令牌）： 也是 OAuth 2.0 中的一种令牌，用于在 Access Token 过期或失效时获取新的 Access Token。
  4. Session Token（会话令牌）： 在传统的 Web 应用程序中，用于标识用户会话的令牌，通常存储在用户的浏览器 Cookie 中。

下文中的token是指JWT，先简单介绍一下JWT

JWT

JSON Web Token（JWT）是一种开放标准（RFC 7519），用于在两个实体之间安全地传递信息。它是一种紧凑且独立的方式，用于在各方之间作为 JSON 对象安全地传输信息。JWT 可以在用户和服务之间传递声明（Claims），以便在各方之间安全地传输信息。这些声明可以被验证，因此接收方可以信任这些声明。

JWT 由三部分组成，每个部分之间使用点（.）进行分隔：

  1. Header（头部）： 包含了两部分信息，声明类型（typ）和使用的签名算法（alg）。
  2. Payload（负载）： 包含了一系列声明（Claims），是实际传输的数据。
  3. Signature（签名）： 对前两部分的签名，以确保数据的完整性和验证发送方的身份。
    • 使用 Base64 编码的 Header 和 Payload 组成一个字符串，用.连接。
    • 使用事先约定好的密钥对这个字符串进行加密，得到签名

用户登录过程：

  1. 用户发起登录请求： 用户提供用户名和密码，发送登录请求到服务器。
  2. 服务器验证用户身份： 服务器验证用户提供的用户名和密码是否匹配。如果验证成功，服务器生成一个包含用户信息的令牌（Token）。
  3. 令牌生成： 生成的Token通常包括以下信息：
    • 用户ID
    • 过期时间（Token的有效期限）
    • 其他必要的信息，如用户角色、权限等
  4. 令牌返回给客户端： 服务器将生成的Token作为响应返回给客户端。
  5. 客户端存储Token： 客户端通常将Token存储在本地，如浏览器的LocalStorage或应用程序的内存中。

后续请求的认证：

  1. 用户发起受保护资源的请求： 用户在登录后，发送请求访问需要身份验证的资源。
  2. Token发送到服务器： 客户端将存储的Token放入请求的头部（通常是Authorization头）中，发送到服务器。
  3. 服务器验证Token： 服务器收到请求后，解析Token，验证其有效性和是否过期。
  4. 访问控制： 如果Token有效，服务器会根据Token中包含的用户信息进行访问控制，决定是否允许用户访问请求的资源。
  5. 响应： 服务器返回请求资源的响应，或者如果Token无效，则返回相应的错误信息。

Session

用户登录过程：

  1. 用户发起登录请求： 用户提供用户名和密码，发送登录请求到服务器。
  2. 服务器验证用户身份： 服务器验证用户提供的用户名和密码是否匹配。如果验证成功，服务器创建一个会话（Session）并在服务器端存储用户的会话信息。
  3. 会话ID生成： 为了唯一标识用户的会话，服务器生成一个唯一的会话ID。这个ID通常是一个长字符串或数字。
  4. 会话信息存储： 用户的会话信息（如用户ID、角色、权限等）被存储在服务器端的一个数据结构中，通常是在内存中或者持久化存储中。
  5. 会话ID返回给客户端： 服务器将生成的会话ID发送给客户端，通常通过设置一个名为JSESSIONID的Cookie。

后续请求的认证：

  1. 用户发起受保护资源的请求： 用户在登录后，发送请求访问需要身份验证的资源。
  2. 会话ID发送到服务器： 客户端将包含会话ID的Cookie放入请求头中，发送到服务器。
  3. 服务器验证会话ID： 服务器收到请求后，根据会话ID查找存储在服务器端的会话信息。
  4. 访问控制： 如果找到会话信息，服务器根据会话信息进行访问控制，决定是否允许用户访问请求的资源。
  5. 响应： 服务器返回请求资源的响应，或者如果未通过身份验证，则返回相应的错误信息。

对比

Session相对来说是比较传统的Web应用解决方案，主要问题有：

1、数据需要存储在服务器中，对于用户量大的系统内存占用很明显

2、对于分布式系统，需要额外的处理来管理Session状态，因为采用默认的负载均衡策略在A服务器登陆后，请求发送到B服务器是需要再次登录的。一般采用的策略有通过路由策略将客户端绑定到固定服务器上，或者将Session状态存储在Redis缓存中。

而Token是基于Session的问题进行改进得来的，它也有问题，即利用了时间换空间，其节省了服务器的内存资源，但是需要耗费CPU去计算token来进行验证。

Sa-token

Sa-Token 是一个轻量级 Java 权限认证框架，主要解决：登录认证、权限认证、单点登录、OAuth2.0、分布式Session会话、微服务网关鉴权 等一系列权限相关问题。相比于Shiro、SpringSecurity 等框架的繁多配置，Sa-Token的API设计简洁优雅的多，可以使用其来作为鉴权框架。文档：框架介绍 (sa-token.cc)

第三方登录认证

该系统有五种登录认证方式，分别为Email认证、Password认证、Sms短信认证、Social社交平台认证、Xcx认证，在登录的请求传进来的时候，需要根据传入请求携带的授权类型grantType调用对应的认证服务。

Social社交平台认证是通过开源项目JustAuth实现，其支持大量第三方平台的OAuth登录。项目地址：https://gitee.com/yadong.zhang/JustAuth

Xcx认证是微信小程序调用授权，这个功能项目中只写了接口，具体实现并没有完成，也不太了解这个认证。
